# Native SIGSEGV Crash During SceneClassifier Initialization

## 问题描述

在 21073cf 之后的版本中，App 启动时在 `MainViewModel.init` 中发生 native 层崩溃：

```
Fatal signal 11 (SIGSEGV), code 1 (SEGV_MAPERR), fault addr 0x0 in tid 13893 (DefaultDispatch)
```

崩溃线程是 `DefaultDispatch`（Kotlin 默认 IO 调度器），对应 `MainViewModel` 的 `viewModelScope.launch(Dispatchers.IO)` 协程。

## 影响版本

- **21073cf 及之前**：正常，图像特征可以正确提取
- **21073cf 之后**：启动崩溃，无法完成图像特征提取

## 根因分析

### 新增的 `SceneClassifier.initialize()` 调用

21073cf 版本的 `MainViewModel.init`：

```kotlin
init {
    viewModelScope.launch(Dispatchers.IO) {
        val ok = clip.init(false)
        _ready.value = ok                          // ← ok 可能为 false，但仍继续执行
        _indexedCount.value = repo.count()
        loadClusters()
    }
}
```

当前版本的 `MainViewModel.init`：

```kotlin
init {
    viewModelScope.launch(Dispatchers.IO) {
        val ok = clip.init(false)
        sceneClassifier = SceneClassifier(extractor)
        sceneClassifier.initialize()               // ← 新增：调用 10 次 encodeText → native
        _ready.value = ok
        _indexedCount.value = repo.count()
        loadClusters()
    }
}
```

### 问题链路

1. `sceneClassifier.initialize()` 对 10 个预设场景标签各调用一次 `extractor.encodeText()`
2. `encodeText()` → `NcnnClip.encodeText()` → `encodeTextNative()` JNI 调用
3. `encodeTextNative()` 调用 `g_text.create_extractor()` 创建 NCNN 文本推理器
4. 如果 `clip.init(false)` 返回 `false`（模型文件加载失败），`g_text` 内部状态为空
5. `g_text.create_extractor()` 返回一个内部指针为空的 `Extractor`
6. 调用 `ex.input("in0", in0)` 时访问空指针 → **SIGSEGV (fault addr 0x0)**

### 为什么 21073cf 没有崩溃？

21073cf 版本没有 `sceneClassifier.initialize()` 调用，`MainViewModel.init` 只做 `clip.init()` + 数据库查询，不涉及 native 推理。图像特征提取只在 `IndexWorker`（独立进程）中执行。

### `clip.init(false)` 可能失败的原因

- assets 中模型文件（`clip_vision.param`、`clip_vision.bin`、`clip_text.param`、`clip_text.bin`）不存在
- 模型文件损坏或格式不匹配
- NCNN 版本不兼容

## 修复方案

### 修复 1：MainViewModel init 失败保护（核心修复）

**文件**：`app/src/main/java/com/example/picsearch/MainViewModel.kt`

```kotlin
init {
    viewModelScope.launch(Dispatchers.IO) {
        val ok = clip.init(false)
        if (!ok) {
            _ready.value = false
            return@launch          // ← 模型加载失败时直接退出，不调用 sceneClassifier
        }
        sceneClassifier = SceneClassifier(extractor)
        sceneClassifier.initialize()
        _ready.value = true        // ← 只在全部成功后才标记 ready
        _indexedCount.value = repo.count()
        loadClusters()
    }
}
```

**原理**：native crash 无法被 Kotlin `try-catch` 捕获，唯一可靠的防护是确保 `clip.init()` 成功后才调用任何 encode 方法。

### 修复 2：initNative 线程安全（防御性加固）

**文件**：`app/src/main/cpp/clip_jni.cpp`

```cpp
#include <mutex>

static std::mutex g_init_mutex;

extern "C" JNIEXPORT jboolean JNICALL
Java_com_example_picsearch_ml_NcnnClip_initNative(...) {
    std::lock_guard<std::mutex> lock(g_init_mutex);  // ← 防止多线程同时初始化
    if (g_inited) return JNI_TRUE;
    // ...
}
```

**原理**：当前版本有 3 个可能并发调用 `initNative` 的路径（MainViewModel、QuickIndexWorker、IndexWorker），全局 `g_visual`/`g_text` 需要互斥保护。

### 修复 3：input/extract 返回值检查（防御性加固）

**文件**：`app/src/main/cpp/clip_jni.cpp`

```cpp
// encodeImageNative
ncnn::Extractor ex = g_visual.create_extractor();
if (ex.input("in0", in) != 0) return nullptr;  // ← 检查返回值

// encodeTextNative
ncnn::Extractor ex = g_text.create_extractor();
if (ex.input("in0", in0) != 0 || ex.input("in1", in1) != 0) return nullptr;  // ← 检查返回值
```

**原理**：原代码忽略 `ex.input()` 的返回值，即使模型加载失败也会继续执行到 `ex.extract()`，导致空指针。

## 变更对比

| 维度 | 21073cf（正常） | 当前版本（崩溃） |
|------|-----------------|-----------------|
| `MainViewModel.init` 调用 sceneClassifier | ❌ | ✅（崩溃源） |
| `MainViewModel.init` init 失败保护 | ❌ | ✅（修复后） |
| `clip_jni.cpp` init mutex | ❌ | ✅（修复后） |
| `clip_jni.cpp` input 返回值检查 | ❌ | ✅（修复后） |

| `clip.init()` 模式 | Vulkan (`true`) | CPU (`false`) |
| `clip_jni.cpp` 核心逻辑 | 无变化 | 无变化 |

## 验证方法

1. 确保 assets 目录下存在以下 4 个模型文件：
   - `clip_vision.param`
   - `clip_vision.bin`
   - `clip_text.param`
   - `clip_text.bin`

2. 安装 APK 后观察 logcat：
   - 出现 `CLIP_DEBUG` 标签的预处理日志 → 模型加载成功
   - 出现 `SIGSEGV` → 模型文件缺失或损坏

3. 快速索引应能正常完成 Top 100 照片的特征提取
